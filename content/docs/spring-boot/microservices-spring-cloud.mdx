# Microservices & Spring Cloud

A microservices architecture splits a large backend application into separate, independent services. Spring Cloud provides a set of tools to coordinate these services, offering solutions for service discovery, gateway routing, and centralized configuration management.

## Core Concepts
Understanding service discovery and API gateways is essential for building distributed microservices.

### Service Discovery: Eureka Server
In dynamic microservices environments, service instances scale up and down, changing their IP addresses continuously. **Eureka Server** acts as a centralized service registry where each microservice registers its IP address and port on startup, allowing instances to find each other dynamically.

### API Gateway Routing: Spring Cloud Gateway
The **API Gateway** acts as the single entry point for all client requests. It handles tasks like security authentication, load balancing, rate limiting, and request routing to specific backend services.

### Centralized Configuration: Config Server
Managing environment configurations across dozens of microservices is complex. **Spring Cloud Config Server** provides centralized configuration management, storing properties in a git repository and exposing them via REST endpoints to microservice clients.

## Code Example

Here is an implementation of a routing configuration inside a Spring Cloud Gateway service:

```yaml
# application.yml gateway routing configurations
spring:
  cloud:
    gateway:
      routes:
        # Route users API requests to user-service
        - id: user_route
          uri: lb://user-service          # Load balance using Eureka registry names
          predicates:
            - Path=/api/users/**
            
        # Route orders API requests to order-service
        - id: order_route
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
```

## Best Practices

*   **Use Eureka Service Registry**: Always resolve service instances using Eureka registry names (e.g. `lb://user-service`) instead of hardcoding IP addresses.
*   **Enforce API Gateway Security**: Validate JWT tokens at the API gateway layer first, before forwarding requests to downstream microservices.
*   **Configure Circuit Breakers**: Use circuit breakers (like Resilience4j) to prevent failures in one microservice from cascading and taking down other services.
*   **Implement Distributed Tracing**: Use tools like Spring Cloud Sleuth or Zipkin to track and debug requests across microservice boundaries.

---

## Common Mistakes

*   **Hardcoding microservice IP addresses**: Defining static URLs inside service code, causing client requests to fail when service instances scale or restart.
*   **Bypassing the API Gateway**: Allowing client browsers to communicate directly with internal microservices, bypassing security gates.
*   **Ignoring distributed log aggregation**: Failing to centralize service logs, making debugging errors across distributed services difficult.

<Callout type="note" title="Load Balancing">
  Spring Cloud Gateway integrates with Spring Cloud LoadBalancer under the hood to distribute incoming requests evenly across active service instances.
</Callout>

---

## Key Takeaways

*   Eureka Server acts as a dynamic registry for microservice instances.
*   Spring Cloud Gateway serves as the single entry and security gate for clients.
*   Decouple service references by using load-balanced registry URLs (`lb://`).
*   Centralize configuration properties using Spring Cloud Config Server.
