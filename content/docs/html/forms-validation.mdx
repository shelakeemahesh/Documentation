# Forms & Validation

HTML forms are the primary mechanism for collecting user inputs and submitting transactions to web applications. Building modern forms requires combining descriptive labels, semantic inputs, and native HTML5 validation constraints to ensure security, usability, and client-side error handling.

## Core Concepts

Understanding form attributes and inputs is essential for handling client submissions.

### The Form Element and Method
Forms are defined using the `<form>` tag, which configures how data is sent:
-   **`action`**: The destination URL where the form data is submitted.
-   **`method`**: The HTTP protocol method (usually `POST` for creations, and `GET` for searches).

### Accessible Form Controls
Every input control must be explicitly linked to a `<label>` element. This is done by matching the label's `for` attribute (in React, `htmlFor`) with the input's `id` attribute. This layout allows screen readers to declare input fields correctly when focused.

### HTML5 Native Validation
HTML5 supports native client-side validation using attributes:
-   `required`: Blocks submit if empty.
-   `pattern`: Validates against a regular expression.
-   `min` / `max`: Enforces ranges on number/date fields.
-   `type="email"` / `type="url"`: Validates format automatically.

## Code Example

Here is a complete, accessible registration form featuring native validations, semantic inputs, and placeholder states:

```html
<form action="/api/register" method="POST" class="p-6 bg-zinc-900 border border-zinc-800 rounded-xl space-y-4 max-w-md">
    <h2 class="text-xl font-bold text-zinc-100">Create Account</h2>

    <!-- Email Field -->
    <div>
        <label for="email" class="block text-sm font-semibold text-zinc-300 mb-1">
            Email Address
        </label>
        <input
            type="email"
            id="email"
            name="email"
            required
            class="w-full p-2 bg-zinc-950 border border-zinc-800 text-zinc-200 rounded focus:border-amber-500 focus:outline-none"
            placeholder="name@example.com"
        />
    </div>

    <!-- Password Field with length limits -->
    <div>
        <label for="password" class="block text-sm font-semibold text-zinc-300 mb-1">
            Password (Min 8 characters)
        </label>
        <input
            type="password"
            id="password"
            name="password"
            required
            minlength="8"
            class="w-full p-2 bg-zinc-950 border border-zinc-800 text-zinc-200 rounded focus:border-amber-500 focus:outline-none"
        />
    </div>

    <!-- Submit Button -->
    <button
        type="submit"
        class="w-full py-2 bg-amber-500 text-zinc-950 font-bold rounded hover:bg-amber-400 transition-colors"
    >
        Submit Registration
    </button>
</form>
```

## Best Practices

*   **Link Inputs to Labels**: Always link inputs to labels using `for` and `id` attributes.
*   **Specify Input Types**: Use specific input types (like `email`, `tel`, `number`) to trigger the correct keyboard layout on mobile devices.
*   **Do Not Disable Submit Buttons**: Avoid disabling the submit button when a form is invalid. Allow users to click the button and trigger native validation error tooltips.
*   **Include autocomplete Attributes**: Add autocomplete tags (like `autocomplete="username"` or `autocomplete="current-password"`) to help password managers autofill fields.

---

## Common Mistakes

*   **Using placeholder text as a label**: Omitting the `<label>` tag and relying only on input `placeholder` text, which disappears once the user starts typing and leaves screen reader users disoriented.
*   **Forgetting to define type="button" on non-submit buttons**: Creating helper buttons (like "Cancel") inside a form without declaring `type="button"`, which defaults to `submit` and submits the form accidentally.
*   **Neglecting server-side validation**: Relying *only* on client-side HTML5 validations. Client validations are easily bypassed by backend updates; always re-validate data on the server.

<Callout type="warning" title="Novalidate Override">
  If you want to handle form validation entirely using JavaScript (e.g. React Hook Form) without displaying native browser error tooltips, add the `novalidate` attribute to the `<form>` tag.
</Callout>

---

## Key Takeaways

*   Form tags require explicit method types and endpoint paths.
*   Link labels to inputs to ensure screen reader accessibility.
*   HTML5 input types (like `email`) handle baseline format checking automatically.
*   Always combine frontend client validation with secure server-side validation.
