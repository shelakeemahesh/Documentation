# SSR Overview (Next.js)

Server-Side Rendering (SSR) in Next.js allows React components to be compiled into static HTML pages on the web server in response to each client request. This architecture improves First Contentful Paint (FCP) times and optimizes search engine optimization (SEO) by delivering search-crawler-friendly markup directly to the client browser, compared to traditional client-side rendering (CSR).

## Core Concepts

Understanding SSR requires dissecting rendering lifecycles and modern hydration phases.

### Client-Side Rendering (CSR) vs. Server-Side Rendering (SSR)
In Client-Side Rendering, the server sends a minimal HTML file (usually containing just a single mount container like `<div id="root"></div>`) alongside a large JavaScript bundle. The browser downloads, parses, and executes this JS to build the DOM, resulting in a blank screen or loading spinner during initial load. 
In Server-Side Rendering, the server executes React code, queries databases, and converts components into an HTML page on the fly. The browser displays this completed HTML immediately, before the JavaScript bundle finishes downloading.

### Hydration: Bringing Pages to Life
HTML generated by the server is static and non-interactive. **Hydration** is the client-side process where React loads the JavaScript bundle, matches the client-side virtual DOM with the server-rendered HTML markup, and attaches event listeners (like `onClick`) to the static elements. Once hydrated, the app behaves like a standard React Single Page Application.

### React Server Components (RSC) vs. SSR
It is important to distinguish between Server Components (RSC) and Server-Side Rendering (SSR):
-   **SSR** is a lifecycle phase where components are rendered to HTML on the server.
-   **RSC** is a component model. Server Components run exclusively on the server, never download to the client, and do not hydrate. Client Components (marked with `"use client"`) run on both the server (compiled to HTML) and the client (hydrated).

## Code Example

Here is a Next.js 14+ App Router configuration showing a Server Component fetching database records directly, passing data down to a interactive Client Component:

```tsx
// 1. Server Component: Runs exclusively on the server (app/products/page.tsx)
import React from 'react';
import { InteractiveFavoriteButton } from './FavoriteButton';

interface Product {
  id: number;
  title: string;
  price: number;
}

// Data fetching in Server Components is direct and asynchronous
async function getProducts(): Promise<Product[]> {
  const res = await fetch('https://api.example.com/products', {
    next: { revalidate: 60 }, // ISR: Cache and revalidate every 60 seconds
  });
  if (!res.ok) throw new Error('Failed to fetch product data');
  return res.json();
}

export default async function ProductsPage() {
  const products = await getProducts();

  return (
    <div className="p-8 space-y-6">
      <h1 className="text-3xl font-extrabold text-zinc-100">Our Products</h1>
      
      <div className="grid gap-4 sm:grid-cols-2 md:grid-cols-3">
        {products.map((product) => (
          <div key={product.id} className="p-4 bg-zinc-900 border border-zinc-800 rounded-xl flex flex-col justify-between">
            <div>
              <h3 className="font-bold text-zinc-200">{product.title}</h3>
              <p className="text-amber-500 font-mono mt-1">${product.price.toFixed(2)}</p>
            </div>
            {/* Interactive Client component nested within Server component */}
            <div className="mt-4 pt-4 border-t border-zinc-800">
              <InteractiveFavoriteButton productId={product.id} />
            </div>
          </div>
        ))}
      </div>
    </div>
  );
}
```

```tsx
// 2. Client Component: Hydrated on the client (app/products/FavoriteButton.tsx)
'use client';

import React, { useState } from 'react';
import { Heart } from 'lucide-react';

export function InteractiveFavoriteButton({ productId }: { productId: number }) {
  const [isFavorited, setIsFavorited] = useState<boolean>(false);

  return (
    <button
      onClick={() => setIsFavorited((prev) => !prev)}
      className={`inline-flex items-center gap-2 text-xs font-semibold px-3 py-1.5 rounded-lg border transition-colors ${
        isFavorited
          ? 'bg-rose-500/10 text-rose-400 border-rose-500/20'

      }`}
    >
      <Heart className={`h-4 w-4 ${isFavorited ? 'fill-current' : ''}`} />
      {isFavorited ? 'Favorited' : 'Add to Favorites'}
    </button>
  );
}
```

## Best Practices

*   **Use Server Components by Default**: Keep components as Server Components to reduce the size of the client-side JavaScript bundle and speed up initial page hydration.
*   **Fetch Close to the Component**: In Next.js App Router, fetch data directly inside the Server Component that requires it instead of fetching at the top-level layout.
*   **Safeguard Browser Globals**: If you must access browser APIs (like `window` or `localStorage`), place them inside `useEffect` blocks or use dynamic imports with SSR disabled.
*   **Implement Streaming with Suspense**: Wrap slow server data fetches in a React `Suspense` boundary to stream the rest of the page layout immediately while the data loads.

---

## Common Mistakes

*   **Accessing browser APIs directly in module scope**: Trying to read `window` or `document` outside a `useEffect` hook in a client component, which throws compilation errors because Next.js pre-renders client components on the server.
*   **Marking all files with "use client"**: Unnecessarily declaring client-side code, which forces Next.js to package the component into the bundle, losing the bundle-splitting benefits of Server Components.
*   **Hydration mismatch warnings**: Rendering dynamic data (like random numbers or local system times) directly in the JSX, resulting in a markup difference between the server HTML and client hydration.

<Callout type="note" title="Solving Hydration Mismatch">
  If you must render client-specific dynamic data (like formatted date strings), disable server pre-rendering for that specific component by importing it dynamically with `ssr: false` in Next.js:
  ```typescript
  const DynamicDate = dynamic(() => import('./DateComponent'), { ssr: false });
  ```
</Callout>

## Key Takeaways

*   SSR compiles React components into raw HTML on the server for each request.
*   Hydration merges the static HTML layout with client-side JavaScript actions.
*   React Server Components do not download or run on the client browser.
*   Use `useEffect` or dynamic imports to handle client-only browser APIs.
