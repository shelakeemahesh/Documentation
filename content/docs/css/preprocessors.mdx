# Preprocessors (Sass/SCSS)

CSS preprocessors (like Sass/SCSS) extend the capabilities of standard CSS by adding scripting features like variables, nested selectors, mixins, and mathematical functions. Preprocessor files compile down to standard CSS files before being served to browser nodes.

## Core Concepts

Understanding preprocessor compilation pipelines and modularity tools is key to scaling large stylesheets.

### SCSS vs. Sass Syntax
-   **SCSS (Sassy CSS)**: Uses the `.scss` extension and features standard brackets and semicolons, making it compatible with vanilla CSS files.
-   **Sass (Indent Syntax)**: Uses the `.sass` extension and relies on indentation and newline spaces instead of braces and semicolons.

### Nested Selectors
Preprocessors allow you to nest child style rules inside parent blocks. This mimics the HTML DOM tree structure and keeps stylesheets organized:
```scss
.card {
    background: #000;
    .title { color: #fff; } /* Selects .card .title */
}
```

### Reusable Logic: Mixins and Extends
-   **Mixins**: Defined using `@mixin` and imported using `@include`. They are functions that output blocks of reusable CSS rules based on passed arguments.
-   **Extends**: Allow elements to share and inherit blocks of declarations, reducing stylesheet size duplication.

## Code Example

Here is an SCSS snippet demonstrating nesting, parent selector references (`&`), and mixin calls:

```scss
/* 1. Define mixin for flex alignment */
@mixin flex-center {
    display: flex;
    justify-content: center;
    align-items: center;
}

/* 2. Style block demonstrating nesting and selectors */
.primary-button {
    @include flex-center; /* Inject mixin styles */
    padding: 10px 20px;
    background-color: #3b82f6;
    color: #ffffff;
    
    /* Nest hover rules using parent reference (&) */
    &:hover {
        background-color: #2563eb;
    }
    
    /* Target nested icon class */
    .btn-icon {
        margin-right: 8px;
        width: 16px;
    }
}
```

## Best Practices

*   **Avoid Deep Nesting**: Limit your selector nesting depth to 3 levels. Deep nesting bloats your compiled CSS and makes overriding rules difficult.
*   **Organize with Partial Imports**: Split styles into modular files (like `_buttons.scss`, `_grids.scss`) prefixed with an underscore, and combine them using `@use`.
*   **Use Mixins for Reusable Layout Blocks**: Use mixins to abstract complex layout patterns (like viewport configurations or flexbox alignments).
*   **Prefer SCSS syntax**: Standardize on SCSS syntax to allow vanilla CSS code to be copied and pasted directly without syntax conversion.

---

## Common Mistakes

*   **Nesting selectors excessively**: Creating selector paths like `.card .body .wrapper .title span a`, which inflates specificity scores and slows down browser rendering.
*   **Forgetting underscores on partial imports**: Omitting underscores on helper files, causing compile pipelines to output them as separate, independent CSS files instead of importing them.
*   **Using @import instead of @use**: Continuing to use the deprecated `@import` rule (which can cause variable namespace conflicts) instead of the modern `@use` standard.

<Callout type="note" title="Compilation Pipeline">
  Browsers cannot read SCSS files directly. They must be compiled into standard CSS using build tools (like Dart Sass, Vite, or Webpack loaders) before deployment.
</Callout>

---

## Key Takeaways

*   Preprocessors add variables, nested structures, and reusable mixins to CSS.
*   SCSS matches vanilla CSS formatting; Sass uses tabs and indentation.
*   Limit nesting depth to 3 levels to keep compiled selectors clean.
*   Organize modular partial files using the modern `@use` import standard.
